3修 bug 必须修模板

只修实例，不修 skill = 没修

手动删了空 CLAUDE.md → 下个 Agent 按 skill 又生成回来

反问 1 — 溯源

这个文件谁让 Agent 生成的？

反问 2 — 验证

改完 skill 后重新跑一遍，
旧产物还会不会复活？

  • 核心观点：
  • 改生成物没用，要改生成它的指令。
  • 两个反问是最实用的落地习惯。
  • 反问 1 Agent 能帮你做（grep skill 文件），反问 2 需要人判断。
  • 解法放口播：
  • 用 PostToolUse hook 自动检测 skill 产物被编辑，
  • 提醒把改动回流到 skill 模板。

  • AI 一旦犯错，先补规则再修代码；
  • 尽量不要靠手写补丁兜底。

5执行模式的 4 次进化

版本模式触发痛点
1.0 单 Agent 顺序执行 3228 行文件撑爆上下文
2.0 人工拆分 + 并行 Agent 每次都要重写提示词
3.0 Skill 诊断 + 提示词生成 用户仍是 orchestrator
4.0 主 Agent 纯编排 + 自动编排 零暂停点自动化
  • 逐行推进。
  • 1.0：最朴素的方式，一个 Agent 从头干到尾，
  • 3228 行文件撑爆上下文，被迫拆分——解决容量。
  • 2.0：并行能跑了，但每次都要重写提示词，纯重复劳动，
  • 于是固化成 Skill——解决重复。
  • 3.0：Skill 能生成提示词了，但用户还是 orchestrator，
  • 每步手动粘贴、判断顺序、处理失败，7~9 个暂停点。
  • 4.0：主 Agent 接管编排，自动派 subagent、自动重跑，
  • 暂停点降到 0——解决暂停。
  • 每次进化都是被痛点逼出来的，不是提前设计的。

3.0 vs 4.0

3.0

  • 执行主体：用户
  • 人工暂停点：7~9 个
  • 失败恢复：用户看报告决定
  • 并行执行：手动开多个 Agent
  • 最终输出：markdown 片段拼接

4.0

  • 执行主体：主 Agent
  • 人工暂停点：0 个
  • 失败恢复：3 次升级重跑 + auto-fix
  • 并行执行：自动编排 自动派 subagent
  • 最终输出：固定 7 段式模板
  • 重点对比暂停点：从 7~9 个到 0 个，这是最直观的进步。
  • 3.0 用户还在复制提示词到新 Agent、手动判断顺序。
  • 4.0 主 Agent 自己编排、派 subagent、失败自动重跑 3 次。
  • 最终 Skill 从"给人读的操作手册"变成了"给 Agent 执行的半程序"。
1.0→2.0 解决容量，2.0→3.0 解决重复，
3.0→4.0 解决暂停。

5 条可复用的原则

原则含义推论
1. 仓库即 Context 让代码库本身成为高质量上下文，不额外构建上下文系统 项目越干净，Agent 越强
2. 技术债是乘法成本 Agent 批量操作时，坏设计乘以所有文件 优先偿还乘法因子最大的债
3. 反馈环路越短越好 PostToolUse hook 让反馈从天级压缩到 3 秒 强类型是编译期质量闸
4. Guides + Sensors Guides 拦截意图，Sensors 检测结果，缺一不可 Sensors 是不依赖模型的硬约束
5. 修 bug 必须修模板 只修实例不修模板，下一轮 Agent 会复活同一份 bug 用 hook 自动提醒回流 skill
  • 逐条讲，每条 30 秒左右。
  • 原则 1-2 用转折点 1 数据做锚点。
  • 原则 3-4 用转折点 2 实验做锚点。
  • 原则 5 用转折点 3 故事做锚点。
  • 原则 3 可举 3 个“编译期质量闸”例子：
  • 1) TypeScript：接口字段变更后，未同步的调用点在 tsc 阶段直接报错，避免运行时 undefined。
  • 2) Java Spring Boot：方法签名或泛型类型不匹配时，mvn/gradle 编译阶段直接失败，问题不会带到接口联调。
  • 3) Rust：所有权/借用不满足时编译不通过，提前拦截数据竞争和悬垂引用。
  • 反例：纯 JavaScript 项目里把 number 当 string 传递，通常要到运行时或线上日志才暴露，反馈延迟更长、修复成本更高。

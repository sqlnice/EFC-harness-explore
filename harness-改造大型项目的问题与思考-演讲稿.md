然后抛问题：

### AI 编程的三次进化：Prompt → Context → Harness（3-4 分钟）

| 阶段 | 核心动作 | 比喻 | 真实体验 |
|------|---------|------|---------|
| **Prompt 时代** | 写精确的指令让 AI 生成代码 | 拉老虎机 —— 提示词写得好就中奖，写不好就废 | 2025 年用 Cursor，基本需要手动告诉它上下文，chat 里生成代码再复制到项目里改 |
| **Context 时代** | 给 AI 足够的上下文，让它理解项目 | 给实习生一本操作手册 | 开始写 CLAUDE.md、给 AI 喂项目结构，产出质量明显上升但仍不稳定 |
| **Harness 时代** | 让代码库本身成为 AI 的反馈系统 | 给工程师配一套自动化流水线 —— 不管谁来写，写错了流水线会拦住 | EFC 改造：tsc/eslint/vitest 三层 sensor + hook 实时反压，AI 写完代码后自己就知道对不对 |

**金句**：

> "你不需要在代码库之外额外构建上下文系统 —— 不需要刻意做 Context Engineering，不需要单独搭 RAG。你需要做的，是让代码库本身成为一个高质量的上下文。**仓库即 Context。**"

---

### 为什么传统协作方式不适合 Agent（2-3 分钟）

"三省六部"式的组织架构 —— 给不同的人分配不同的职责、用流程文档约束行为 —— 在 Agent 时代不适用：

- Agent 没有"记忆上一轮犯了什么错"的能力，每次启动都是一张白纸
- 传统方式依赖**人的经验积累**来保证质量；Agent 方式需要**系统的反馈机制**来保证质量
- 以前思考成本高（涉及架构评审、人员协调），现在迭代快速 —— "不要怕，经验是迭代出来的"

**金句**：

> "以前保证代码质量靠的是'找对的人'；现在保证代码质量靠的是'建对的闸门'。人会疲劳、会遗忘、会偷懒，但编译器 + lint 不会。"

### 转折点 1：技术债务会被 Agent 指数级放大

**对应时间**：Day 1，阶段一~三

**故事**：

改造第一天，让 Agent 做一件看起来很简单的事 —— 把 services/ 目录下的 .js 文件迁移成 .ts 并加上类型标注。

结果发现：
- `request` 方法的签名是双泛型 `<T, R>`，第一个 `T` 纯占位，调用时必须写 `request.get<any, Base.IResponseList<Tenant>>`
- 启用 `no-explicit-any` 后，这个占位 `any` 立刻变成了**每个 service 函数都报错**
- 一个两年前为了"方便"留下的签名设计，在 Agent 批量迁移时变成了几百处连锁报错

**数据**：

| 指标 | 数值 |
|------|------|
| 消除的 `: any` | 560 处 → 0 |
| 消除的函数入参 any | 1906 处 |
| 精确类型替换 | 2150 次 |
| 涉及 service 文件 | ~110 个 |

**洞察**：

> "技术债务被 Agent 指数级放大 —— 这是 52 天里最反直觉的发现。
>
> 人写代码时，一个糟糕的签名只影响你正在写的那个函数。但 Agent 是批量操作的 —— 一个坏签名乘以 200 个 service 函数，就是 200 处报错。
>
> **人时代的技术债是线性成本，Agent 时代的技术债是乘法成本。**"

### 转折点 2：反馈环路越短，Agent 迭代效率越高

**对应时间**：Day 2，阶段四~六

**故事**：

测试写完、hook 配好之后，做了一个实验 —— 故意让 Agent 引入一个类型错误来测试 hook 是否生效。

有趣的事情发生了：让它"把类型改成 any"，它**拒绝了** —— 因为它读到了 CLAUDE.md 里的禁止规则。这说明第一层防护（Guides / 前馈）在工作。

为了测试第二层（Sensors / 反馈），不得不绕过 Guides，用一种不触发禁止词的方式引入错误。结果 PostToolUse hook 立刻捕捉到了 tsc 报错，Agent 自动回退修复。

**反馈延迟对比**：

| 机制 | 触发时机 | 反馈对象 | 反馈延迟 |
|------|---------|---------|---------|
| CLAUDE.md 禁止规则 | Agent 读到上下文时 | Agent 自己 | 0（前馈） |
| PostToolUse hook | Agent 每次编辑文件后 | Agent 自己 | ~3 秒 |
| pre-commit hook | `git commit` 时 | 开发者 | 分钟级 |
| Code Review | PR 提交后 | 开发者 | 小时~天级 |

**洞察**：

> "Guides 和 Sensors 形成两层防护。Guides 在 Agent 动手之前拦截违规意图，Sensors 在 Agent 动手之后检测违规结果。**两层都需要，因为 Guides 可以被忽略，但 Sensors 不会说谎。**"

> "反馈延迟从天级压缩到秒级，Agent 的迭代效率不是线性提升 —— 是质变。"

补充：
- pre-commit hook 是给**人**的最后防线，PostToolUse hook 是给**Agent**的实时反压
- vitest reporter 设为 `dot`（沉默设计：成功无输出，失败才报错）—— 反馈环路的"降噪"也很重要

### 转折点 3：修 bug 必须同时修产生 bug 的模板

**对应时间**：Day 3，阶段八

**故事**：

用 `/health` skill 对项目做了一次配置栈审计，发现 5 个问题并逐一修复。修完之后回头看 `efc-harness` skill 本身 —— 发现**skill 的指令里还在教 Agent 犯我们刚刚修掉的错**。

具体例子：
- 刚删掉了 sms-web 下一个空的 CLAUDE.md（确认是反模式）
- 但 skill 里的提示词 F 还在写"为子应用创建 CLAUDE.md"
- 如果下一个用户调用这个 skill，Agent 会忠实地再生成一份我们刚删掉的文件

**洞察**：

> "在有 skill / rules / guide 这类 meta 配置层的项目里，**修 bug 必须同时修'产生 bug 的指令模板'**。只修实例不修模板等于没修 —— 下一轮 Agent 会按旧模板忠实地复活同一份 bug。"

**落地习惯**：

改完任何配置后反问两件事：

1. 这个 bug 是 Agent 按某条指令生成的吗？那条指令还在哪里？
2. 如果下一个 Agent 严格按新指令执行，会不会重新生成我刚删掉的东西？

> "第二步 Agent 做不了 —— 它没有'上次删掉了什么'的记忆。**这是目前人必须补位的节点。**"

### 转折点 4：Agent 能写 AST 和状态机来帮你扫描死代码

**对应时间**：Day 4-5，实战案例

**故事**：

改造过程中让 Agent 自己派发 subagent 扫描项目里的死代码。它不是简单地 grep —— 它写了一个 AST 解析脚本，用状态机逻辑去追踪 import 链路，找出真正没被引用的 symbol。

**数据**：

| 项 | 数量 |
|---|------|
| 扫描文件 | 3173 |
| 扫描 import 语句 | 4239 |
| 命中死 symbol | 34 |
| 自动清理（A+B 类 + 校验通过） | 5 |
| Category C（需人工确认） | 6 |
| Used-but-missing（symbol 不存在但业务实际调用） | 29 |

**洞察**：

> "Agent 不是只会写 CRUD —— 它能写 AST 解析器、状态机、批量重构脚本。**关键不是 Agent 能不能做到，而是你的代码库能不能让它做到。**"

> "保证代码质量不是靠眼睛去读代码，**是靠编辑器爆红 + lint 这个积累很长时间的工程化体系来强制校验**。Harness 的本质就是把这个体系从'给人用的'扩展到'给 Agent 也能用的'。"

### 转折点 5：执行模式的 4 次进化

**对应时间**：Day 3-6，阶段十一~十二

**故事**：

改造过程中，人和 Agent 的协作方式本身也在进化。不是提前设计的，而是**每一次被痛点逼到墙角后才发生的**：

| 版本 | 模式 | 触发痛点 | 解决方式 |
|------|------|---------|---------|
| **1.0** | 单 Agent 顺序执行 | 一个 3228 行的文件把上下文撑爆 | 拆成多个并行 Agent |
| **2.0** | 人工拆分 + 并行 Agent | 每次都要重写提示词，重复劳动 | 固化成 Skill（诊断器 + 提示词生成器）|
| **3.0** | Skill 诊断 + 提示词生成 | 用户仍是 orchestrator，每步都要手动粘贴、判断顺序、处理失败 | 主 Agent 接管编排，0 暂停点 |
| **4.0** | 主 Agent 纯编排 + DAG | —（当前形态）| 诊断 → 自动派 subagent → 3 次升级重跑 → 7 段式最终汇报 |

**3.0 vs 4.0 对比**：

| 维度 | 3.0 | 4.0 |
|------|-----|-----|
| 执行主体 | 用户（复制提示词到新 Agent） | 主 Agent（纯编排，自动派 subagent） |
| 人工暂停点 | 每步骤后 1 个（7~9 个） | 0 个 |
| 失败恢复 | 用户看报告决定重试/跳过 | 3 次升级重跑 + auto-fix 白名单 |
| 并行执行 | 用户手动开多个 Agent | DAG 并行点自动派多个 Agent |
| 最终输出 | 各步骤 markdown 片段拼接 | 固定 7 段式模板 |

**洞察**：

> "1.0→2.0 解决**容量**问题，2.0→3.0 解决**重复**问题，3.0→4.0 解决**暂停**问题。
>
> 每一次进化都是被实际痛点逼出来的 —— 如果一开始就冲着 4.0 设计，你会塞满大量'理论上有用但实际不需要'的机制。**从简单开始，失败后再改进。**"

> "最终 Skill 从'一份给人读的操作手册'变成了'一份给 Agent 执行的半程序' —— 每条指令都考虑了'如果 Agent 严格按字面执行会发生什么'，并给出退化路径。"

## 第三部分：方法论提炼

### 5 条可复用的原则

每条原则一句话定义 + 一个证据锚点。

---

**原则 1：仓库即 Context**

> 不要在代码库之外额外构建上下文系统。让代码库本身成为高质量的上下文 —— 强类型、清晰的目录结构、零死代码、有意义的命名。Agent 找到正确上下文的成本越低，产出质量越高，消耗的 token 越少。

证据：EFC 改造后，Agent 不再需要手动喂上下文。类型文件、lint 规则、CLAUDE.md 禁止事项 —— 这些都是代码库里已有的信息，Agent 自己读取就够了。

推论：及时清理技术债务不再是"有空再做"的事 —— 它直接决定了 Agent 的产出质量。**项目越干净，Agent 越强；项目越脏，Agent 越蠢。**

---

**原则 2：技术债务是 Agent 的乘法成本**

> 人写代码时，一个坏设计影响你正在写的那一处。Agent 批量操作时，一个坏设计乘以它触及的所有文件。技术债务从线性成本变成乘法成本。

证据：一个双泛型签名设计 × 200 个 service 函数 = 200 处连锁报错。

推论：在引入 Agent 之前，优先偿还那些会被"批量触及"的技术债 —— 公共签名、基础类型、共享工具函数。这些是乘法因子最大的地方。

---

**原则 3：反馈环路越短，Agent 迭代效率越高**

> 把错误反馈从"天级"（Code Review）压缩到"秒级"（PostToolUse hook），Agent 的迭代效率发生质变。

证据：PostToolUse hook 让 Agent 每次编辑文件后 3 秒内就知道有没有 tsc/eslint 报错，自动回退修复。

推论：强类型语言（TypeScript、Go、Proto）天然提供编译期反馈，是 Agent 最好的搭档。**强类型是编译期质量闸** —— 它把大量错误从运行时前移到编译时，而编译时恰好是 Agent 能自动响应的时间窗口。

---

**原则 4：Guides 和 Sensors 互补，缺一不可**

> Guides（CLAUDE.md / rules）在 Agent 动手前拦截意图，Sensors（tsc / eslint / test / hook）在 Agent 动手后检测结果。Guides 可以被忽略，Sensors 不会说谎。

证据：让 Agent "把类型改成 any"，Guides 拦住了；绕过 Guides 引入错误，Sensors 捕捉到了。

推论：不要只写 CLAUDE.md 就觉得"Agent 知道规矩了"。指令遵循不是 100% 的 —— 特别是上下文变长、任务变复杂时，Guides 会被"遗忘"。**Sensors 是不依赖模型能力的硬约束，是最后一道防线。**

---

**原则 5：修 bug 必须同时修产生 bug 的模板**

> 在有 skill / rules / guide 这类 meta 配置层的项目里，只修实例不修模板等于没修。下一轮 Agent 会按旧模板忠实地复活同一份 bug。

证据：删了空的 CLAUDE.md，但 skill 里还在写"为子应用创建 CLAUDE.md"。

推论：每次修完一个 Agent 生成的问题，反问自己："这个 bug 是按某条指令生成的吗？那条指令还在哪里？" **这一步目前必须由人来做 —— Agent 没有'上次删了什么'的记忆。**

### 对 Opus 4.6 的体感

从 2025 年 Cursor 到 2026 年 Claude Code + Opus 4.6 的个人体验对比：

| 维度 | 2025 Cursor | 2026 Claude Code + Opus 4.6 |
|------|------------|---------------------------|
| 上下文 | 需要手动告诉它看哪些文件 | 自己探索项目结构，自己定位相关代码 |
| 批量操作 | 一次改一个文件，人工串联 | 自己派发 subagent 并行处理，一次改几十个文件 |
| 代码能力 | 能写业务代码 | 能写 AST 解析器、状态机、批量重构脚本 |
| 工具编写 | 需要人写辅助脚本 | 自己写一次性工具脚本来解决问题，用完就扔 |
| 协作模式 | Chat → 复制到项目改（拉老虎机） | Skill 编排 → DAG 执行 → 零暂停点自动化 |

> "不是 AI 变聪明了所以我们可以偷懒 —— 是 AI 变聪明了所以我们可以把精力从'写代码'转移到'建闸门'。**人的角色从 Generator 变成了 Evaluator。**"

**Commit 类型分布**：

| chore | 99 | 32% | 依赖升级、配置统一 |
| fix | 80 | 26% | Vite 运行时、路由、SCSS、类型 |
| refactor | 66 | 21% | JS→TS 迁移、any 消除 |

**修复的 80 个 Bug 主要类别**：Vite 开发环境（Sass/webpack 冲突/WebSocket proxy）、类型系统（TS 4.9 升级/shims 声明污染）、路由（NavigationDuplicated 静默/权限字段对齐）、SCSS 语法适配、构建配置、共享层 drift 同步、格式化级联修复

### 结尾

> "Prompt 时代我们教 AI 写代码。Context 时代我们教 AI 理解项目。Harness 时代我们不再教 AI —— 我们建一个系统，**让 AI 自己知道有没有做对**。
>
> 这个系统不是什么新东西，就是我们积累了很多年的工程化体系 —— 类型系统、lint、测试、CI。唯一的区别是，现在它的反馈对象不只是人，还有 Agent。"

### 后续探索：从"改造完成"到"持续健康"

EFC 的 7 天改造解决了**代码级**的健康检查（Sensors + Hooks + Back-pressure）。但健康检查是一个三层架构，后两层还有很大的探索空间：

```
┌─────────────────────────────────────────────┐
│  第三层：系统级                 自愈反馈循环   │  ← 未来
│  ┌───────────────────────────────────────┐  │
│  │  第二层：流程级    CI/CD + AI 审查     │  │  ← 下一步
│  │  ┌─────────────────────────────────┐  │  │
│  │  │  第一层：代码级    Hooks + 验证  │  │  │  ← 已完成
│  │  └─────────────────────────────────┘  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

**已完成：第一层 —— 代码级**

EFC 改造已覆盖的部分：
- 计算型 Sensors（tsc / eslint / vitest）
- PostToolUse Hook（Agent 每次编辑后自动验证）
- PreToolUse Hook（拦截对 generated 文件和基建配置的直接编辑）
- Loop 检测（同一文件修改超过 5 次时警告 Agent 重新评估方案，防止"末日循环"）
- Back-pressure（pre-commit 三道关卡 + 沉默设计）

**下一步：第二层 —— 流程级**

EFC 当前使用 Jenkins 做构建部署，但质量守护完全靠 Husky pre-commit 本地执行，Jenkins 管线中没有集成 tsc / eslint / vitest 等检查。没有 E2E 测试。这意味着第二层的关键不是"搭 CI"，而是"把已有的质量关卡从本地上移到 Jenkins"：

| 方向 | 做什么 | 预期收益 |
|------|--------|---------|
| Build-Verify 四步流程 | 在 CLAUDE.md 中明确 Plan → Build → Verify → Fix 循环，Agent 标记任务完成前必须跑通验证 | 防止 Agent "自认为完成"但实际没验证 |
| Jenkins 管线增强 | 将目前纯本地的 pre-commit（lint + tsc + test）上移到 Jenkins，每次推送自动执行 | 本地守护可以被绕过（`--no-verify`），Jenkins 不可跳过 |
| AI 代码审查 | PR 触发审查 Agent：代码质量 / 安全性 / 依赖项扫描 | 从"人工 review"变成"Agent 先审 + 人工终审" |

**未来：第三层 —— 系统级自愈闭环**

第三层对 EFC 来说距离较远，但值得了解方向。CREAO 的生产案例已经证明了这条路径的可行性 —— 完整的自愈反馈循环可将部署频率从"两周一次"提升到"每天 3-8 次"：

```
定时健康检查
  → AI 查询监控系统
  → 生成系统健康摘要
  → 分诊引擎聚类错误（多维度评估严重程度）
  → 自动创建工单（含日志样本 / 受影响用户 / 排查方向）
  → Agent 修复
  → CI 管线验证
  → 分诊引擎重新验证
  → 自动关闭工单
```

**EFC 走到这一步需要补的前提**：

| 前提 | EFC 现状 | 差距 |
|------|---------|------|
| 自动化测试 | vitest 2,987 用例（纯单元测试） | 缺 E2E 测试覆盖 |
| CI/CD 流程 | Jenkins 已有，但未集成质量关卡 | 需要将 tsc/eslint/vitest 接入 Jenkins 管线 |
| 线上监控 | cms-web 是产品自身的监控模块，项目本身无可观测性 | 需要为项目本身接入监控 |
| 任务管理 | 无外部工具 | 需要引入可被 Agent 读写的任务系统 |
| 系统架构 | qiankun 微前端，9 子应用已统一 harness | 基本就绪 |

> "Harness 不是一次性工程。7 天的改造让我们站上了第一层台阶 —— 代码级的健康检查。后面还有流程级和系统级两层要爬。但每一层的核心思想都一样：**让反馈环路覆盖所有会影响产出质量的环节，让机器替人做确定性判断。**"
